## 1. 参考サイトから学ぶ原理(観察事実 / 推測 / すぴたろうへの応用 を分離)

参考: サンリオ・シナモロール / ちいかわ / PUPPET SUNSUN の各公式サイト。
**ロゴ・キャラ・イラスト・配色再現・装飾配置・アイコン・アニメ素材・文章・フォント・コード・独自レイアウトは一切模倣しない。** 抽出するのは設計原理のみ。

| # | 観察事実(一般的に見られる UI パターン) | 推測(なぜそうしているか) | すぴたろうへの応用案(独自表現に変換) |
|---|---|---|---|
| 1 | キャラクターを画面中央〜上部に大きく置く | 初見でキャラを主役だと伝えるため | 現ヒーローを維持。将来もキャラ画像を情報より先に見せる |
| 2 | プロフィールを小さな区画に分けて視覚化 | 長文を読ませず拾い読みさせるため | 「宇宙カプセル/観測窓」風の項目カード(§8)。円形の固有配置は真似ない |
| 3 | 項目をタップ → 詳細が出る | 情報を段階開示し画面を軽く保つため | ボトムシート方式(§7)。中身とトリガーは独自 |
| 4 | キャラが小さく常時動く | 「そこにいる」感を出すため | 既存の浮遊 + 控えめな傾き(§10)。動きの種類・量は独自に抑制 |
| 5 | イラストをギャラリーで見せる | ファンの滞在時間と愛着を上げるため | /gallery(§9)。画像は正式素材のみ、空状態から始める |
| 6 | キャラ同士の関係を見せる | 世界観の広がりを感じさせるため | relationships データ枠だけ用意(§6)。架空キャラは作らない |
| 7 | フッターで SNS へ自然誘導 | 回遊の出口を明確化するため | FOLLOW ブロック(§12)。ロゴ素材は使わずテキスト+安全属性 |
| 8 | モバイル縦スクロールで発見が続く構成 | 片手で読み進められるため | セクションを縦一列で維持(既存方針の延長) |

## 2. 模倣を避ける固有表現(すぴたろうの世界)

配色はサイト既存トークン(白 / ベビーブルー / 深い青 / 控えめな紫 / 夜空の濃色)を継続。他社の配色・円配置・キャラクター造形・書体を再現しない。すぴたろう固有のモチーフ = **宇宙 / 丸いもの / 泡・星のかけら / 観測窓 / 独自の言葉(おぱぁ 等)**。これらを自前でデザインする。

## 3. サイトコンセプト(再定義)

> 人間には言葉がよく分からない、かわいい宇宙生物たちの世界を、見て、触って、少しずつ知っていくサイト。

設計上の優先: **短文 / 表情 / 絵 / 動き / タップ反応 / 視覚的発見 / 余白 / 音声非依存**。長文プロフィールを中心に置かない(既存 `content/profile.ts` の「意味の解説を付けない」方針を全 Phase で堅持)。

## 4. 情報構造

将来像: ホーム / キャラクター(→ すぴたろう / 将来の生き物)/ ギャラリー / お知らせ / プロフィール / SNS。現在キャラは 1 体なので、空のキャラ一覧を大きく見せない。

- **案A**: 現行 `/profile` をすぴたろう詳細として維持、複数化時に構造変更
- **案B**: `/characters` + `/characters/supitaro` を先に用意(複数対応に強いが今はページ増・移行コスト)

**判定: 現段階は案A を採用(ルート変更は今回しない)。** 根拠 = キャラ 1 体で `/characters` 一覧は中身が薄く UX 上メリットが出ない。ただしデータ層(§5)は**最初から複数キャラ対応の形**で作り、ルート移行は Phase 5 でキャラ 2 体目が確定した時に案B へ引き上げる。この「データは複数対応・URL は単数維持」により、後の移行差分を最小化できる。

## 5. キャラクターデータモデル

複数キャラを見据えつつ、**現在はすぴたろう 1 体のみ登録・架空キャラは作らない**。既存 `SiteImage`(`config/site.ts`)と `content/profile.ts` を再利用して接続する。

```ts
type Localized = { ja: string; en: string }; // en は Phase 4 まで空文字許容
type CharacterTrait = { key: string; label: Localized; value: Localized };
type CharacterRelationship = { slug: string; label: Localized }; // 相手 slug 未確定の間は空配列
type GalleryItem = { /* §9 参照 */ };

type CharacterProfile = {
  slug: string;                    // "supitaro"
  name: Localized;
  summary: Localized;              // 短い紹介(既存 intro を移送)
  images: { main: SiteImage; profile: SiteImage };
  words: string[];                 // 独自の言葉。意味解説は持たせない
  traits: CharacterTrait[];        // 下記候補から確定分のみ
  gallery: GalleryItem[];
  relationships: CharacterRelationship[]; // 現在は []
  isPublished: boolean;
};
```

trait 候補(**確定分のみ登録、未確定は創作しない**): キャラクター紹介 / 出身 / 性格 / 好きなもの / 苦手なもの / 特技 / 大きさ / 話す言葉 / 地球でしていること / 出会った生き物。
現状確定しているのは 紹介・出身(宇宙)・好きなもの(丸いもの)・話す言葉(おぱぁ 等)・地球でしていること(こっそりゲーム配信)のみ。性格 / 苦手 / 特技 / 大きさ / 出会った生き物は **未確定 → 空。ユーザー確定後に追記**。

移行方針: `content/profile.ts` → `content/characters/supitaro.ts` へ Phase 2 で無損失移送。`en` は Phase 4 まで空文字を許容し、空なら ja フォールバック。

## 6. タップ式プロフィール UI(比較と推奨)

| 観点 | モーダル中央 | **ボトムシート** | アコーディオン | インライン展開カード |
|---|---|---|---|---|
| 360px 読みやすさ | 中(幅が窮屈) | 高(全幅) | 高 | 高 |
| 片手操作 | 低(中央/×が上) | **高(下部で親指圏)** | 中 | 中 |
| 閉じやすさ | Esc/× | **Esc/×/下スワイプ/背景** | 再タップ | 再タップ |
| キーボード | 要フォーカストラップ | 要フォーカストラップ | 容易 | 容易 |
| スクリーンリーダー | dialog role | dialog role | 展開状態通知 | 展開状態通知 |
| No-JS | × | × | △(details で可) | ○(常時表示に退避) |
| URL 共有 | × | ハッシュで可 | × | ○(アンカー) |
| 長文対応 | 中 | 高(スクロール) | 高 | 高 |
| 実装複雑度 | 中 | 高 | 低 | 低 |
| 複数キャラ対応 | ○ | ○ | ○ | ○ |

**推奨: モバイルはボトムシート(短〜中文の項目詳細に最適)、長文セクションはインライン展開。** No-JS 代替 = §で必須要件(下記)。

受け入れ要件(実装時): 項目全体を 44px 以上の `<button>` にする / 開いた内容に見出し / Escape で閉じる / 閉じるボタン / 閉じた後トリガーへフォーカス返却 / 背景スクロールロック / `prefers-reduced-motion` でスライドイン無効化 / 色だけで「押せる」を示さない(枠線 + アフォーダンス記号併用)。

**No-JS 代替方針(必須)**: プロフィールの全文は**サーバー出力の HTML に最初から含める**。ボトムシートは JS 有効時の「拡大ビューア」に過ぎず、JS 無効時は各項目の詳細がそのままインライン表示される(details/summary か、常時展開の段落)。= 既存 FadeIn と同じ「SSR は全部見える、JS は装飾」の設計原則を踏襲。これにより情報消失ゼロ。

## 7. プロフィール項目の視覚設計(2〜3 案)

サンリオの円形は**原理のみ参照**(円形ボタン化は必須でない)。モバイル要件: 2 列以内 / 文字を小さくしない / 円内に長い日本語を詰めない / 項目名 2 行以内 / 画面端に余白 / 押せると分かる / 縦に伸ばしすぎない。

- **案1「宇宙カプセル」**: 角丸長方形カード + 左に小さな丸アイコン + 項目名。2 列グリッド。長い日本語に強い。**← 推奨**
- **案2「観測窓」**: 上部が半円の窓型カード。装飾性は高いが日本語の折返しが窮屈になりやすい
- **案3「泡」**: 円形バブル。可愛いが円内テキスト制約が §要件に反しやすく、長い値で破綻

**推奨: 案1(宇宙カプセル型カード)。** 既存の rounded-2xl カード資産を流用でき、44px 確保・2 行制約・2 列を最も無理なく満たす。丸モチーフはアイコン部分にだけ効かせて世界観を担保。

## 8. デザインギャラリー

目的: イラストを見せる / 愛着 / SNS 外の世界観 / 将来は他キャラも。

```ts
type GalleryItem = {
  id: string;
  characterSlug: string;      // "supitaro"
  image: SiteImage;           // 既存型を再利用(src 空はプレースホルダー)
  title?: Localized;
  description?: Localized;
  publishedAt?: string;       // "YYYY-MM-DD"
  category?: string;
  isPublished: boolean;
};
```

設計判断:
- **`/gallery` を独立ページにする**(トップには載せない or 数枚の抜粋のみ将来検討)。プロフィール内には出さず責務分離
- グリッド: モバイル 2 列 / タブレット 3 列 / PC 3〜4 列。縦長・横長・正方形が混在するため **`aspect` を固定枠 + `object-contain`** で切れ・レイアウト崩れを防ぐ(顔を切らない既存方針と一致)
- 拡大表示は Phase 3 で軽量ライトボックス(No-JS 時は画像単体リンクにフォールバック)/ alt 必須 / 遅延読み込み(hero 以外) / WebP 最適化 / モバイルはスワイプ可(JS 有効時のみ、無効時は縦並び)
- 権利: 画像は**正式素材のみ**。仮素材(temporary artwork)はギャラリーに出さない。右クリック保存の技術的抑止はしない(実効性がなく UX を損なう)方針とし、フッター/ページに利用条件の一文を将来添える
- **現時点は画像素材ゼロ → プレースホルダーを量産しない。空状態のみ設計**: 「イラストは準備中です。すこしずつ増えていきます。」を中央に控えめ表示 + 宇宙モチーフの装飾枠(既存プレースホルダーのトーン)。`isPublished` が無い/全件 false のとき空状態を出す

## 9. キャラクターの動き(優先順位と技術比較)

「そこにいる」感を独自表現へ。**第一段階の上限**: CSS/SVG 中心 / 常時動作は 1〜2 種 / GIF 必須にしない / 音自動再生なし / Canvas・WebGL・Live2D なし / 低性能端末で動く / reduced-motion 停止 / 読解を邪魔しない / layout shift なし。

優先順位:
1. **ゆっくり浮遊**(既存・維持)
2. **ときどき小さく傾く**(±2〜3°、数秒に一度。CSS keyframes、transform のみ)
3. タップで表情変化 + 「おぱぁ」吹き出し(Phase 2、JS 必要)
4. 星/泡がゆっくり流れる背景(Phase 3、CSS、密度控えめ)
5. まばたき・尻尾(Phase 3 以降、SVG の正式キャラ確定後)

技術比較:

| 方式 | 長所 | 短所 | 採否 |
|---|---|---|---|
| CSS animation | 軽量・reduced-motion 容易・layout shift 回避容易 | 複雑演出は不得手 | **主軸** |
| SVG (SMIL/CSS) | ベクター・拡縮自由・正式キャラと相性 | 正式 SVG が必要 | Phase 3 で併用 |
| Motion(導入済) | 物理/ジェスチャに強い | JS 依存・バンドル増 | タップ反応など**必要箇所のみ** |
| GIF | 制作が楽 | 容量大・reduced-motion で止められない・alt 制約・レトロ画質 | **不採用** |

**GIF は採用しない。** 常時動作は 1〜2 種(浮遊 + 傾き)に留める。

## 13. 多言語対応(方式比較と推奨)

今回、翻訳・ルーティングは実装しない。

| 観点 | 案A `/ja/` `/en/` | 案B `/`+`/en/` | 案C 同一 URL クライアント切替 |
|---|---|---|---|
| SEO | ◎ 別 URL で明確 | ○ | × 同一 URL で分離不可 |
| canonical/hreflang | ◎ 素直 | ○ | × |
| URL 共有 | ◎ 言語固定で共有 | ◎ | × 言語が伝わらない |
| 静的生成 | ◎ | ◎ | △ |
| No-JS | ◎ | ◎ | × 切替不能 |
| 翻訳漏れ | 顕在化しやすい | しやすい | 隠れる |
| App Router 相性 | ◎(セグメント) | ◎ | △ |
| 既存 URL 影響 | 大(全部 /ja/ 化) | **小(日本語は / のまま)** | なし |
| お知らせ翻訳 | 全件必要 | en は用意分のみ | — |

**推奨: 案B(日本語を既定として `/` のまま、英語を `/en/` 接頭辞)。**
決定事項:
- **日本語を既定言語**にする / **既存 URL(`/` `/profile` `/news`)を維持**(SEO・共有リンク・被リンクを壊さない。案A の全 URL 変更は Stage A 公開後には不利)
- 英語未翻訳コンテンツの扱い = `/en/` では**用意できた英語のみ公開**、未翻訳ページは英語版を出さない(欠けた hreflang にする。中途半端な機械翻訳を出さない)
- 言語切り替えの配置 = フッターの Language(§11)。将来ヘッダーにも小さく
- **ブラウザ言語による自動リダイレクトはしない**(共有 URL の言語固定を尊重、No-JS でも破綻しない)
- **機械翻訳をそのまま公開しない**。管理された英語文章を用意する方式を優先
- hreflang/canonical は本番ドメイン確定後(別 Sprint)に整備
